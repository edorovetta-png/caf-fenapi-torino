Architettura — Quale approccio?

Il brief definisce già stack e struttura. La differenza sta in come organizziamo il codice e gestiamo multi-utente + ruoli.

A

Monolite Semplice (Raccomandato)

Un singolo progetto Vite con React Router. Supabase gestisce auth + RLS per i ruoli. TanStack Query per cache e sync tra operatori.

Pro

  • Setup rapido — un solo progetto da gestire
  • Supabase RLS gestisce i permessi lato DB (sicuro)
  • TanStack Query + Supabase Realtime per sync multi-utente
  • Deploy semplice su Vercel (quando serve)

Contro

  • Scanner e dashboard nello stesso bundle (mitigabile con lazy loading)
  • Ruoli gestiti solo via RLS + check frontend
B

App Separata per Scanner

Due app: una PWA leggera solo scanner (mobile), una dashboard completa (desktop). Stesso Supabase backend.

Pro

  • PWA scanner ultra-leggera e veloce
  • Separazione netta mobile vs desktop

Contro

  • Due codebase da mantenere
  • Duplicazione logica (auth, types, Supabase client)
  • Complessità doppia per un team piccolo
C

Monolite + Supabase Edge Functions

Come A, ma con logica business (calcolo totali, validazione ordini) spostata in Edge Functions server-side.

Pro

  • Logica business protetta lato server
  • Più sicuro per calcoli critici (prezzi, totali)

Contro

  • Setup più complesso (Deno runtime per Edge Functions)
  • Overhead per un'app interna su rete WiFi
  • Debugging più difficile

💡 Raccomandazione: Opzione A

Per un'app interna multi-operatore, il monolite semplice è la scelta migliore. Supabase RLS fa il lavoro pesante sui permessi, TanStack Query gestisce la sincronizzazione, e React.lazy() tiene il bundle leggero su mobile. Nessun motivo per complicare l'infrastruttura.
